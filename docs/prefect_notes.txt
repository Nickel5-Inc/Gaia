Initialization & Setup
├── init_(self, args)
│    └── Constructs the GaiaValidator object: sets up tasks, the database manager, the HTTP client, and graceful shutdown hooks.
├── setup_neuron(self) → bool
│    └── Loads environment variables, configures the Substrate endpoint, wallet, and hotkeys, syncs metagraph, and confirms successful connection.
└── main(self)
     ├── Calls setup methods (e.g., neuron, database).
     ├── Spawns periodic tasks (geomagnetic, soil, scoring, etc.).
     └── Waits for shutdown events to gracefully cancel tasks and clean resources.

Event Handling & Graceful Shutdown
├── _signal_handler(self, signum, frame)
│    └── Captures system signals (SIGTERM, SIGINT, etc.) and initiates a graceful shutdown.
├── _initiate_shutdown(self)
│    └── Cancels active tasks, closes open resources, and ensures cleanup steps are complete before exiting.
├── cleanup_resources(self)
│    └── Performs comprehensive cleanup (temp file removal, resetting database operations, cleaning task states, collecting garbage).
└── recover_task(self, task_name)
     └── Attempts to recover a hung or timed-out task by forcing cleanup and re-initializing the necessary resources.

Main Loop & Periodic Tasks
├── main_scoring(self)
│    └── Periodically calculates and sets on-chain weights from recent task scores, applies decay/normalization, and submits them to the network.
├── _calc_task_weights(self)
│    └── Decays historical scores and generates final weights used by main_scoring.
├── status_logger(self)
│    └── Logs system and block info periodically, including active node count and time since last weight update.
├── update_miner_table(self)
│    └── Fetches updated miner metadata (hotkey, coldkey, IP, etc.) from the metagraph and stores in the database.
├── handle_miner_deregistration_loop(self)
│    └── Checks for miners deregistered from the chain, cleans database entries, and recalculates scores.
├── check_for_updates(self)
│    └── Periodically checks if code updates should be pulled and applied.
├── start_watchdog(self) / stop_watchdog(self)
│    └── Starts or stops the watchdog that monitors tasks for excessive usage or lock-ups.
├── _watchdog_loop(self) / _watchdog_check(self)
│    └── Core watchdog logic: checks for timed-out tasks and ensures metagraph sync.
├── _check_resource_usage(self, current_time) / _check_task_health(self, current_time)
│    └── Continuously collects CPU/memory usage and verifies task time limits.
└── update_task_status(self, task_name, status, operation=None)
     └── Updates the internal health structure to track the state (active, idle, error) of tasks and logs resource changes.

Types of scheduling
+++++++++++++++++++

- IntervalScheduleInterval:


* Interval:

Allows you to set a specific time interval between flow runs.
Can be specified in seconds or using a timedelta object.

Example: interval=600 (runs every 10 minutes)

* Cron:

Uses standard cron syntax for more complex scheduling.
Allows for schedules based on specific times, days, months, etc.

Example: cron="0 * * * *" (runs every hour)

* RRule (iCal Recurrence Rules):

Provides a flexible way to create repetitive schedules.
Can handle complex logic like "last weekday of each month" or "every other day of the week".

Example: rrule="FREQ=WEEKLY;BYDAY=MO,WE,FR;UNTIL=20240730T040000Z"

All these schedule types can be combined with additional options like:

* timezone: To specify a particular time zone for the schedule.
* anchor_date: For interval schedules, to set a starting point for the schedule.

## Core Components

### Task Structure
- Tasks are atomic units of work
- Can be synchronous or asynchronous
- Support retries, timeouts, and triggers
- Can be parallelized using Dask

### Flow Structure
- Flows organize tasks into workflows
- Support parallel execution
- Handle dependencies between tasks
- Can use different task runners (e.g., DaskTaskRunner)

### Parallel Processing with Dask
Located in `gaia/parallel/`:
1. **Configuration** (`config/settings.py`):
   - Define node types (VALIDATOR, MINER)
   - Define task types (IO_BOUND, CPU_BOUND, MEMORY_BOUND, MIXED)
   - Configure resources per task type
   - Set worker and thread counts

2. **Execution** (`core/executor.py`):
   - ParallelExecutor class for Dask integration
   - Handles cluster management
   - Manages parallel task distribution
   - Automatic resource cleanup

3. **Utilities** (`utils/helpers.py`):
   - @parallelize decorator for simple functions
   - @parallel_flow for Prefect flows
   - @parallel_task for Prefect tasks
   - Batch processing helpers

### Usage Examples

1. Parallel Task:
```python
@parallel_task(
    node_type=NodeType.VALIDATOR,
    task_type=TaskType.CPU_BOUND,
    retries=3
)
async def process_data(data):
    return analyze(data)
```

2. Parallel Flow:
```python
@parallel_flow(
    node_type=NodeType.VALIDATOR,
    task_type=TaskType.IO_BOUND,
    name="process_batch"
)
async def process_batch(items):
    results = []
    for item in items:
        result = await process_data(item)
        results.append(result)
    return results
```

3. Direct Parallel Processing:
```python
async with get_task_runner(TaskType.MIXED) as runner:
    futures = [
        runner.submit(process_func, items[i:i+10])
        for i in range(0, len(items), 10)
    ]
    results = [await f for f in futures]
```

## Resource Management

### Task Types and Resources
1. IO_BOUND Tasks:
   - More threads per worker (2+)
   - Good for network/file operations
   - Example: downloading data, API calls

2. CPU_BOUND Tasks:
   - Single thread per worker
   - Good for computations
   - Example: data processing, scoring

3. MEMORY_BOUND Tasks:
   - Fewer workers, more memory per worker
   - Good for large data operations
   - Example: model inference

4. MIXED Tasks:
   - Balanced resources
   - Good for combined operations
   - Example: preprocessing + inference

### Configuration Example
```python
DEFAULT_CONFIGS = {
    TaskType.IO_BOUND: LocalResourceConfig(
        resources={'io': 1},
        adapt_kwargs={'minimum': 1, 'maximum': 8}
    ),

        resources={'cpu': 1},
        adapt_kwargs={'minimum': 1, 'maximum': 4}
    )
}
```

## Query Flow Example

```python
@flow(name="query_miners")
async def query_miners(regions):
    """Query miners in parallel for predictions."""
    async with get_task_runner(TaskType.IO_BOUND) as runner:
        futures = [
            runner.submit(process_region, region)
            for region in regions
        ]
        return [await f for f in futures]

@task(name="process_region")
async def process_region(region):
    """Process a single region with parallel scoring."""
    async with get_task_runner(TaskType.CPU_BOUND) as runner:
        predictions = await get_predictions(region)
        scores = await runner.submit(calculate_scores, predictions)
        return {"region": region, "scores": scores}
```
